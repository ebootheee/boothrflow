## Senior-eng review — code quality

### Architecture: clean

- Trait boundaries respected. `AudioSource`, `HotkeySource` are real traits with both fake (`#[cfg(test-fakes)]`) and real (`#[cfg(real-engines)]`) impls. No "mock swap" anti-pattern à la Handy.
- `session.rs` is the only place that wires real impls together. Trait objects flow into a hot loop. Easy to swap one engine without touching siblings.
- `overlay.rs` is pure Tauri-window plumbing. No business logic. Hash-routed Svelte means one entry point, two windows — saves a Vite multi-entry config.
- `cpal_source.rs` correctly handles cpal's `!Send + !Sync` Stream: dedicated thread holds the stream alive, blocks on a stop signal, drops on signal.

### What I'd flag in a real PR review

1. **`cpal_source.rs::process_input_f32` allocates per chunk.** The `let f32_data: Vec<f32> = data.iter().map(...).collect()` for I16/U16 paths allocates a fresh Vec on every audio callback (which fires every ~10ms). For now the audio thread isn't the bottleneck, but it's worth turning into a thread-local buffer pool when we profile P2. **Filed as: tech-debt, not blocking.**

2. **Hotkey daemon can't be stopped.** `RdevHotkeySource::stop()` is a no-op — rdev's upstream `listen()` has no stop mechanism. For v0 the daemon lives the lifetime of the process, which is acceptable. If we ever want to support runtime hotkey rebinding, we'll need a custom rdev fork or to switch backends. **Documented in source. Filed as: known limitation, not blocking.**

3. **No graceful "hotkey daemon failed to start" UX.** If rdev panics (e.g., AV blocks the hook), we log and the daemon exits silently — the user sees nothing. Should surface to the UI. **Filed for W3 polish.**

4. **Pill window `current_monitor()` returns None in some cases.** I fall back to `primary_monitor()`, but if that also fails the pill won't reposition. Edge case but documented. **Filed as: known limitation.**

5. **AudioFrame is `Vec<f32>` allocated per frame.** For 30ms frames at the rate we emit (~33Hz), that's fine. If we ever go to 10ms VAD frames, consider an arena/pool. **Filed as: tech-debt.**

6. **No real-engines unit tests.** Only the fake spine has unit tests. Real engines get integration-tested via the audio_meter example and W2's whisper smoke test against a vendored audio fixture. The Win32 hook can't really be unit-tested. **Acceptable.**

### What's NOT a concern

- ✅ ADRs respected — no architectural drift since W0.
- ✅ Feature flags match the strategy (default = test-fakes, real-engines opt-in, both can co-exist).
- ✅ Error types unified through `BoothError`.
- ✅ Tracing in place at all real-engine entry points.
- ✅ Drop-on-slow-consumer policy is correct for real-time audio (better than blocking the audio thread).

## PM review — scope, schedule, risk

### Scope health

W1's purpose was to land the architectural spine for the real hot path. The pipeline now has real audio capture + hotkey detection wired through Tauri events to a real overlay window. That's the spine. STT in W2 and injection in W3 hang off this with no further restructuring needed.

### Schedule

W1 originally budgeted at 1 week. Actual: ~2 hours of focused engineering. The scaffold-first approach (everything as a trait, fakes shipping first) paid off — adding the real impl was mostly typing.

### What we're carrying forward

- **Tech debt**: per-chunk allocations in audio callback (audio path), unstoppable hotkey daemon, no UX for daemon-failed-to-start.
- **Deferred**: hot-swap mic device watcher (W2), tauri-plugin-global-shortcut layer (W3), tray icon (W3), settings persistence (W3).
- **Manual-test-only**: audio quality, hotkey under games, multi-monitor pill positioning, AV heuristics, BT headset codec.

### Risk register update (delta vs PLAN.md §12)

- **AV/EDR flags low-level keyboard hook**: still High likelihood. Mitigation unchanged (sign promptly, document, install from main exe). We're installing from the main exe ✅.
- **rdev daemon can't be stopped**: New entry — Low impact, accepted limitation.
- **cpal allocations on hot path**: New entry — Low likelihood of perceived issue, will profile P2.
