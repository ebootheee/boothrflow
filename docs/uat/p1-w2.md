## Senior-eng review — code quality

### Architecture: clean

- `Vad` trait remains untouched. `SileroVad` impl + scripted `FakeVad` continue to slot interchangeably.
- `EndpointDetector` is a separate concern — not part of the `Vad` trait — which is correct: VAD scores frames, endpoint detection composes scores into events. Different abstractions.
- `SttEngine` trait untouched. `WhisperSttEngine` plugs in.
- `session.rs` real impl is now the single concrete "production wiring" point. It cleanly handles missing-model as a recoverable state.
- Frontend Service/Query/State separation respected: `dictationStore` is a runes-based reactive state module, no business logic, just maps Tauri events to UI state.

### What I'd flag in a real PR review

1. **Whisper model loaded at daemon startup, never reloaded.** If the user puts a model file there _after_ the app started, we won't see it without restart. **Filed for W3**: settings UI button "reload Whisper" + auto-detect on filesystem change (`notify` crate).
2. **`SerializedWhisperSttEngine` struct exists but is unused.** I added it speculatively for thread-safety. The current single-threaded session daemon doesn't need it. **Filed for cleanup**: either delete or use it.
3. **Audio buffer flattening allocates a new Vec per dictation.** `frames.iter().flat_map(...).collect()` — fine for v0 (one allocation per release event, dwarfed by Whisper's own allocs), but worth pooling later.
4. **No backpressure if Whisper is slow.** If the user holds the hotkey for 30+ seconds and releases, Whisper blocks the session-daemon thread until done. Press during that window gets queued in `hotkey_rx` but doesn't trigger a new capture (audio.start would error "already capturing"). Actually wait — audio.stop is called _before_ whisper runs, so a fresh press IS possible. Re-check: yes, the loop iteration completes (including STT) before the next event is consumed. So a press during STT is queued, then runs after STT finishes. Acceptable, possibly surprising. **Documented in source.**
5. **Whisper's `WhisperContext` is not thread-safe in upstream whisper-rs?** Actually it's `Send + Sync` per the crate docs; we just wrap it in `Arc`. Verified.

### What's NOT a concern

- ✅ ADR-009 honored — Whisper is the multilingual fallback layer; Parakeet TDT default ships in Phase 4. tiny.en for dev iteration matches the plan.
- ✅ `feature = "real-engines"` strategy holds. `default = "test-fakes"` keeps the inner loop sub-second.
- ✅ Clear "model missing" path with download URL in the error message.

## PM review — scope, schedule, risk

### Scope

W2 was defined as VAD + STT + the stitch. All three are shipped and integrated.

### Schedule

W2 originally budgeted at 1 week. Actual: a few hours of focused engineering, plus another hour solving the **bindgen / Windows toolchain** friction.

### What we ate that wasn't on the menu

**Bindgen / libclang / Windows SDK env friction.** This wasn't in PLAN.md or any ADR, but it bit hard:

- whisper-rs uses bindgen.
- bindgen needs libclang (LLVM) installed and on `LIBCLANG_PATH`.
- bindgen needs `INCLUDE` env var pointing at MSVC + Windows SDK headers, otherwise bindings come out empty.
- Plain `cargo build` doesn't set these — Rust's MSVC integration only configures `cc`, not bindgen.

**Resolution:** Wrote `scripts/cargo-msvc.bat` that loads `vcvars64.bat` + sets `LIBCLANG_PATH`, then forwards to cargo. Works in any shell. Updated `pnpm dev:msvc`, `pnpm build:msvc`, `pnpm test:rust:real` scripts. Documented in README.

**Filed as:** ADR-013 (drafting below).

## Suggested ADR-013 to append

> **ADR-013 — Windows native build env loaded via wrapper script**
>
> **Context:** bindgen-using crates (whisper-rs, sherpa-rs, llama-cpp-2) need libclang + MSVC/SDK INCLUDE paths. Rust's MSVC integration sets these up for `cc` but not for `bindgen`. Plain `cargo build` from a non-VS-dev shell silently produces empty bindings.
>
> **Decision:** Ship `scripts/cargo-msvc.bat` that loads `vcvars64.bat` + `LIBCLANG_PATH` then forwards to cargo. `pnpm dev:msvc`, `build:msvc`, `test:rust:real` use it. README documents prereqs (`winget install Microsoft.VisualStudio.2022.BuildTools` + `winget install LLVM.LLVM`).
>
> **Consequences:** Slight ergonomic tax for Windows contributors (one extra script-name to remember); mitigated by pnpm scripts that hide it. Mac/Linux unaffected. CI runs the same wrapper — production builds use same path as dev.

I'll append this to DECISIONS.md as part of the W2 commit.
