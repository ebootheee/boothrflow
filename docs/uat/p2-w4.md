## What landed

- **`LlamaCppLlmCleanup`** (`src/llm/llama.rs`) — `llama-cpp-2` 0.1.145 wrapping llama.cpp's bundled C++. Process-global `LlamaBackend` via `OnceLock`; model loaded once at daemon startup; fresh `LlamaContext` per call (avoids self-referential lifetimes, ~ms cost).
- **Greedy sampling, `n_ctx=2048`, max 256 output tokens.** Greedy because cleanup is deterministic-ish and we want reproducible output for testing.
- **System prompt** that does one job: add punctuation + capitalization, split run-ons, preserve words exactly. Per-style instructions append to the system prompt (Formal / Casual / Excited / Very Casual / Raw).
- **Skip-LLM heuristics**: `Style::Raw` → passthrough, `LLM not loaded` → passthrough, utterance < 6 words → passthrough. The W3 winning cases ("This is a test.", "No fucking way.") all skip the LLM and stay fast; only the run-on case ("This is so cool this works so well holy shit") triggers cleanup.
- **Style picker now reaches Rust.** FE settings store calls `set_dictation_style` Tauri command on every dropdown change; Rust holds an atomic `CURRENT_STYLE` that the daemon reads before each LLM call.
- **`pnpm download:llm`** — fetches Qwen 2.5 1.5B Instruct Q4_K_M GGUF (~1GB) by default. Tiers: `pnpm download:llm 0.5b` (faster), `pnpm download:llm 3b` (better).
- **New event**: `dictation:formatted` — emitted whenever LLM output differs from raw. Carries `{ raw, formatted, style, llm_ms }`.

## Pipeline now

```

  → audio capture (cpal, 32ms 16kHz frames)
```

## Senior-eng review

### Architecture — clean

- `LlmCleanup` trait stable; both fake and real impls slot in. Pipeline doesn't know whether it's a real LLM or `FakeLlmCleanup` returning canned output.
- `LlamaBackend` global is correctly process-singleton via `OnceLock`. No way to leak a second init.
- Model held in `Mutex<LlamaModel>` even though the current daemon loop is single-threaded. Cheap insurance for future where settings reload triggers re-load on a different thread.
- Context created per-call instead of held alongside model. Simpler than self-referential lifetime gymnastics; <1ms overhead.
- Style enum now `Copy + repr(u8)` so it round-trips through an `AtomicU8` for the FE→Rust style picker without needing channels.

### Things I'd flag in a real PR review

1. **No prompt cache reuse across calls.** Every `cleanup()` re-tokenizes the system prompt + re-prefills the KV cache. The system prompt is ~150 tokens and identical between calls within the same style; pre-computing the prefix KV state would shave ~100-200ms per call. Filed for P2 W5 polish — first measure on real hardware.

2. **Output bytes accumulated over `String::from_utf8_lossy`**. Multi-byte tokens that cross UTF-8 boundaries can be dropped at the seam. Doesn't matter for English; will bite for CJK. Filed for P3 multilingual.

3. **`MAX_OUTPUT_TOKENS = 256`**. If the model gets stuck in a loop ("the the the…"), worst case is 256 tokens at ~30 tok/s = 8s of latency. No early-stopping on repetition yet. Filed for v0.2.

4. **No abort-on-hotkey-while-LLM-running.** If the user holds Ctrl+Win again mid-cleanup, the new press is queued but doesn't preempt. Acceptable for v0; user just waits ~1s.

5. **Style → Rust sync is fire-and-forget.** If the `set_dictation_style` invoke fails (shouldn't, but), the FE thinks the change took effect and the daemon doesn't. No retry/rollback. Acceptable for v0; surface in toast on failure for v1.

6. **`tracing::info!("llm cleanup: \"{raw}\" → \"{text}\"")`** logs both raw and formatted. Privacy-sensitive in production builds; gate behind a flag or downgrade to debug-level for release builds.

### What's NOT a concern

- ✅ Passthrough fallback is honest — if LLM fails for any reason (model missing, decode error, OOM), we paste the raw transcript with a tracing::error, never silently drop or block.
- ✅ Skip-LLM heuristic correctly preserves the W3 fast-path for short utterances.
- ✅ ADR-009 / ADR-011 honored (Qwen 2.5 family, llama-cpp-2 in-process).

## PM review — what to ask the user

When you A/B test, focus on:

1. **The run-on case from W3**: dictate "this is so cool this works so well holy shit" — does the LLM output split into proper sentences with punctuation? That was the failure case W4 was designed to fix.

2. **Single-sentence speed**: dictate "this is a test" — does it still feel sub-second? It should skip the LLM entirely (4 words < 6 threshold).

3. **Style picker**: change Style dropdown from Casual to Formal, dictate, then to Very Casual, dictate. Same words; do you see different formatting?

4. **Edge case**: dictate something the LLM would want to "fix" beyond punctuation. Does it preserve your words, or does Qwen paraphrase? If it paraphrases, the prompt needs tightening.

5. **Latency**: terminal will show `llm cleanup (X ms): "raw" → "formatted"`. That `X` plus Whisper's `Y ms` = total LLM-pipeline cost. Worth a number.

## Recommendation

**Ship to W5.** P2 W5 is **app-context detection** (`GetForegroundWindow` + UI Automation to detect Slack vs Gmail vs IDE) + per-app default styles. That makes the style picker auto-apply based on where you're focused — no manual dropdown. After that, P2 W6 is dictionary + custom AI transformations.

If LLM output quality on your voice is meaningfully worse than tiny.en's already-good baseline, that's a prompt-tuning problem and we can iterate — bring transcripts.
