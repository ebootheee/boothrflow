# Wave 6 Phase 1 bench — STT + LLM swap evaluation

**Date:** 2026-05-21 (two rounds — first pre-`think:false` fix, second post-fix)
**Hardware:** Apple M4 Max, 64 GB unified memory
**Capture:** `2026-05-10T04-32-07.651313+00-00-claudefordesktop.wav` — 204 seconds (3:24) of technical product-planning monologue. Heavy proper-noun density (`GreenPoint`, `CLAUDE.md`, `devops.md`, `gpp.dev`, `shadcn`, `Supabase`, `MCP`, `workers.dev`, `CloudFlare`, single sign-on). Exactly the OOV-heavy content that exercises every model's weak spots.

This is the bench run that closed out the [model survey](../research/model-survey-2026-05.md) recommendations — turned the IFEval / LibriSpeech-WER predictions into real numbers on a real capture, against the existing defaults. The first round caught a procedural error (Qwen3 hybrid thinking-mode wasn't suppressed → false-negative bench); the second round, after the fix landed, is the controlled comparison.

| LLM 7B-class        | `qwen2.5:7b`                 | ✅ **Keep**       | qwen3:8b is 2.6× slower with marginal quality regression (over-edits on Light); qwen3:4b is 9.4× slower and unreliable under bench load (Ollama drops connections during back-to-back swaps) |
| LLM 1.5B-class      | `qwen2.5:1.5b`               | ✅ **Keep**       | qwen3:1.7b is 2× slower AND adds "The cleaned text is as follows:" preambles + paraphrases first-person into third-person, violating Light's "no preambles, no paraphrasing" rules           |

The iron rule for adoption is **net improvement on both speed and quality, or neutral in one with net improvement in the other**. Every candidate fails that rule even after the thinking-mode handling was fixed.

### Why Nemotron lost so badly on this content

Nemotron Speech Streaming en-0.6b's NeMo model card explicitly says: _"not recommended for word-for-word / incomplete sentences"_ and _"out-of-vocabulary words…not likely to be recognized."_ This 204 s capture is the exact worst case:

- 20+ proper nouns concentrated in 3 minutes (GreenPoint, CLAUDE.md, devops.md, gpp.dev, shadcn, Supabase, MCP, GitHub, CloudFlare, workers.dev, single sign-on…)
- Long uninterrupted technical monologue (Nemotron is trained for short streaming chunks)
- Domain-specific abbreviations the streaming model never saw in training

For **short PTT bursts** the live-app test confirmed Nemotron handles fine ("Testing testing tester" decoded perfectly in 413 ms). For long-form technical content, it falls apart. The streaming benefit Nemotron offers — live preview during PTT — isn't measurable in batch decode; that's gated on Wave 6 Phase 1.5 wiring chunked accept_waveform + partial polling in the production hot path.

**Decision:** keep Nemotron as a selectable alternate (it's already in the picker), do not promote it to default.

## Why Qwen3 lost across the board

All three Qwen3 candidates failed the iron rule even after the thinking-mode handling was fixed. Three category-level findings:

### 1. `think: false` doesn't actually stop the thinking

Empirically: a trivial `"hi"` prompt to qwen3:4b with `think: false` set returns 929 completion tokens and a 3,848-character `reasoning` field. The flag controls **where reasoning lands** (separate `reasoning` field vs polluting `content` with `<think>...</think>` blocks), not **whether reasoning happens**. The latency tax — 2-3× compared to a non-thinking model of similar size — is paid regardless.

For dictation cleanup, where the actual transformation is mechanical (drop disfluencies, add punctuation, paragraph breaks), Qwen3's reasoning trace is pure overhead. The bench numbers above are with `think: false` set in every request body.

### 2. The 1.7B size is too small to resist meta-instruction execution

The dictation includes the phrase _"develop a plan for how we update this MCP."_ qwen3:1.7b read that as an instruction:

> The cleaned text is as follows:
>
> The project was a good start but no longer works for our purposes. **The MCP should be updated to create a one-click on-ramp...**

It rewrote the speaker's first-person intent ("I want to turn this MCP into a one-click on-ramp") into a third-person directive ("The MCP should be updated to"). That's the same "executes meta-instructions" failure that retired Assertive style on 2026-05-10. qwen3:8b at 8× the size was robust enough to resist; 1.7B wasn't.

### 3. qwen3:4b is unreliable under bench / sustained-load conditions

Of 8 cleanup attempts during the bench, 7 failed with `error sending request for url` (reqwest-side connection drop) and 1 completed in 112 seconds. Direct curl with the exact bench payload completes in 25 seconds with clean output — so the model is healthy, but Ollama drops connections during back-to-back qwen3:4b calls under the multi-model-resident memory pressure of the bench scenario (qwen3:1.7b + qwen3:8b + qwen3:4b + whisper-base + parakeet + nemotron all in VRAM simultaneously on M4 Max).

Production usage (single dictation at a time, ~5-30 s between calls) would be more like the direct-curl scenario and probably work fine — but the bench evidence says qwen3:4b's reliability envelope is tighter than qwen2.5:7b's by a wide margin.

## Cold-load adjustment

Survey-recommended swaps look different once cold-load time is factored in. For a tool used dozens of times a day, the **first-PTT-after-boot** latency matters as much as warm decode:

| Combo                                              | STT cold | STT decode | LLM cold | LLM cleanup | Total cold-start |
| -------------------------------------------------- | -------: | ---------: | -------: | ----------: | ---------------: |
| `whisper:base.en` + `qwen2.5:1.5b`                 |   ~70 ms |      1.4 s |     ~3 s |       3.6 s |       **~8.1 s** |
| `whisper:base.en` + `qwen3:1.7b`                   |   ~70 ms |      1.4 s |   ~3.5 s |       7.3 s |      **~12.3 s** |
| `whisper:base.en` + `qwen2.5:7b` (current default) |   ~70 ms |      1.4 s |    ~10 s |      12.0 s |      **~23.5 s** |
| `whisper:base.en` + `qwen3:8b`                     |   ~70 ms |      1.4 s |    ~13 s |      30.9 s |      **~45.4 s** |
| `parakeet:0.6b-v2-int8` + `qwen2.5:7b`             |   ~2.5 s |       17 s |    ~10 s |      12.0 s |      **~41.5 s** |
| `nemotron:0.6b-int8` + `qwen2.5:7b`                |   ~3.5 s |       24 s |    ~10 s |      12.0 s |      **~49.5 s** |

The 7B-vs-1.5B cold-start gap (~15 s) is the dominant axis. **qwen2.5:1.5b on this content was indistinguishable from qwen2.5:7b on quality and 3.3× faster**, making it the actual Pareto winner. We're not flipping the default on a single capture — the 2026-05-09 graded bench (per CHANGELOG) put qwen2.5:7b in the default slot for quality reasons across multiple captures — but the picker labels should highlight the cold-start advantage so users on slower machines can self-select.

## What the survey predicted vs what we got

| Survey prediction                                                    | Bench reality                                                                                                                                                                                       |
| -------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `qwen3:4b` Pareto-dominates `qwen2.5:7b` (smaller AND higher IFEval) | 9.4× slower than qwen2.5:7b at median (when it completes at all — 1/8 success under bench load)                                                                                                     |
| `qwen3:8b` is "quality stretch" if tok/s is at least neutral         | tok/s is **not** neutral — 2.6× slower; quality slightly worse (over-edits on Light style)                                                                                                          |
| `qwen3:1.7b` is "direct successor to qwen2.5:1.5b"                   | 2× slower **and** adds preambles + paraphrases first-person to third-person — quality regression on Light style spec                                                                                |
| Nemotron streaming via sherpa-onnx is the highest-value swap         | Wave 6 Phase 1 wiring shipped (model loadable, transcribes), but on this OOV-heavy content the batch quality is materially worse than base.en. Streaming-partial benefit requires Wave 6 Phase 1.5. |

The lesson: **IFEval scores predict instruction-following on benchmark prompts, not real-world dictation cleanup behavior** — especially when the cleanup prompt contains content that looks like task descriptions (which voice dictation often does — "let's draft an email about X", "develop a plan for Y") and the model is small enough or hybrid-mode enough to execute the embedded "instructions" instead of preserving them. And **hybrid Instruct/Thinking mode is a 2-3× latency tax that no flag fully removes** on the current Ollama setup.

## Round-1 vs round-2 comparison

The first round (pre-`think:false` fix) produced a false negative for Qwen3: qwen3:4b couldn't complete a single cleanup, qwen3:1.7b hallucinated an entire implementation plan (likely thinking-mode reasoning leaking into `content`), and the conclusion was framed as "Qwen3 broken." Round 2 (post-fix) is the controlled comparison and gives Qwen3 a fair shake — and even with thinking handled correctly, the Qwen3 family still loses on every axis. The conclusion stands; the framing is now defensible.

Fixes that landed between rounds (commits `6dde9ee` then re-bench):

- `build_system_prompt` appends `/no_think` (Qwen3 control directive; ignored by qwen2.5/llama/etc.)
- `ChatRequest` sends `"think": false` (Ollama OpenAI-compat extension; reasoning goes to a separate `reasoning` field instead of polluting `content`)
- `REQUEST_TIMEOUT` bumped from 30 s to 120 s so hybrid-mode reasoning generation doesn't drop connections on long inputs
